# NK推协小程序 · 运营与体验优化计划（plan.md）

> 本计划聚焦「运营增长」与「用户体验」两条线，不含底层架构/安全（那部分已在服务层迁移、事务幂等加固中完成）。
> 创建于 2026-06-09。优先级：P0 = 高价值低成本，建议优先；P1 = 中等；P2 = 锦上添花。

---

## 一、已具备的基础（避免重复造轮子）

- 服务层封装：`miniprogram/services/*` 全量接入，页面不直连云函数。
- 统一请求：`utils/request.js`（超时/重试/幂等键/飞行去重/登录态静默重登）。
- 上传封装：`utils/upload.js`（选图 + 大小校验 + 云存储 + 错误兜底）。
- 订阅消息：`utils/subscribe.js` + 报名成功 / 兑换通知 / 活动开始提醒（定时触发）。
- 主题系统：`utils/theme.js`（blue / gold 双主题，存 `user_settings.theme`）。
- 分享：核心内容页已配 `onShareAppMessage` + `onShareTimeline`。
- 三态：各页已有 加载 / 空 / 错误（含重试），文案已统一。
- 通用组件：`empty-state`、`common/loading`、`section-card`、`tag-badge`、`detective-card-item`、`cloudTipModal`。

## 二、运营方向（增长 / 留存 / 转化）

### P0-1 订阅消息矩阵补全
- 现状：报名成功、兑换通知、活动开始提醒已上线。
- 待补模板（需在公众平台配置后给出模板 ID）：
  - **委托被领取通知** → 发布者；**委托奖励发放通知** → 领取者。
  - **借阅状态变更通知**（已确认借出 / 已归还）→ 借阅人。
  - **每日谜题发布提醒** → 订阅用户（配合定时触发）。
- 落地要点：复用 `utils/subscribe.js` 的授权封装 + 各云函数 `sendSubscribeMessage` best-effort 下发；新增模板 ID 统一登记在 `subscribe.js` 的 `TEMPLATES`。

### P0-2 签到 / 连续答题激励
- 每日谜题已有「连续答对」统计，可扩展为「连续答题打卡」体系：连续 N 天奖励额外积分。
- 价值：显著提升日活与次日留存。
- 落地：puzzle 云函数答题成功后写/读 `streak`，达标发奖励积分 + 写 `points_log`；首页展示「连续打卡 X 天」。

### P1-1 积分商城运营增强
- 商品「限时 / 限量 / 新品」标签；兑换记录「待领取」状态的取货码与二维码。
- 「积分不足差 X 分」引导去做谜题/委托补分（已存在 `short_points` 字段，可加跳转）。
- 后台支持商品上下架、库存预警。

### P1-2 活动运营闭环
- 活动「即将满员 / 已满员」标签与候补名单（waitlist）。
- 活动结束后「签到/参与确认」→ 发参与积分（attended 状态已预留）。
- 活动详情页分享带活动卡片（路径带 `activity_id` 直达）。

### P1-3 邀请 / 裂变
- 分享路径带邀请人 `user_id`，被邀请人首次登录给双方积分。
- 排行榜分享卡片突出「我的排名」，刺激回流。

### P2-1 消息中心
- 站内「我的通知」列表（报名/兑换/委托/借阅状态变更聚合），弥补订阅消息「一次性」的局限。

### P2-2 运营位 / 公告
- 首页 banner 轮播（关联推荐内容/活动）；系统公告（复用 `system_settings` 或新增 `announcements`）。

## 三、体验方向（流畅 / 一致 / 友好）

### P0-3 三态组件化收敛
- 现状：各页手写 loading/empty/error markup + 各自 WXSS，已有 `empty-state`/`common/loading` 组件但未广泛采用。
- 方案：渐进式替换为统一组件（一次一页，配合视觉回归），消除文案/样式漂移，降低后续维护成本。
- 注意：属重构，需逐页验证，不一次性全量替换。

### P0-4 骨架屏 / 加载体验
- 列表类页面（活动、兑换、借阅、推荐、排行）首屏用骨架屏替代纯 spinner，降低「白屏焦虑」。
- 落地：新增 `skeleton` 组件，按卡片结构渲染占位。

### P1-4 交互细节打磨
- 统一按钮 loading/disabled 态（部分已做）；表单即时校验提示。
- 列表上拉加载更多的「没有更多了」终态文案统一。
- 关键操作（兑换/发布委托/取消报名）二次确认弹窗风格统一（复用 `common/modal`）。
- 下拉刷新动画与节流统一。

### P1-5 图片与上传体验
- 推荐内容封面、兑换商品图后台改为 `utils/upload.js` 直传（替代手填 URL）。
- 列表图片懒加载 `lazy-load` + 默认占位图（部分已有 default 资源）。
- 头像/封面上传加裁剪比例提示。

### P1-6 主题与可访问性
- 主题切换入口在个人中心做显式开关（当前靠 `user_settings.theme`）。
- 检查 gold/blue 两套主题下所有页面对比度与可读性。
- 字号适配（响应系统大字体）。

### P2-3 空数据引导
- 各空态从「暂无数据」升级为「行动引导」（如借阅空态 → 去看看剧本杀；委托空态 → 发布第一个委托）。已部分具备，可全面梳理文案。

### P2-4 性能
- 分包预下载（`preloadRule`）按用户路径优化首进速度。
- 长列表虚拟化（如完整排行榜 100+ 条）。
- 图片走 CDN/压缩，控制单页体积。

## 五、建议执行顺序

1. **第一批（P0，1~2 周）**：订阅消息矩阵补全（委托/借阅/谜题）→ 签到激励 → 骨架屏。
2. **第二批（P1）**：积分商城 + 活动运营闭环 → 三态组件化收敛 → 后台导出筛选。
3. **第三批（P2）**：消息中心、邀请裂变、性能与可访问性打磨。

> 说明：订阅消息相关项需先在微信公众平台配置模板并提供模板 ID；导出/定时类需确认云函数定时触发器与权限配置。每批完成后建议在开发者工具真机预览 + 灰度验证再推广。
